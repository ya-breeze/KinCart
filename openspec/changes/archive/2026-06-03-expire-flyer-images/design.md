## Context

KinCart stores two categories of flyer image files on disk:

1. **`uploads/flyer_pages/<shop>/<shard>/*.jpg`** — raw page images downloaded from retailer websites, referenced by `FlyerPage.LocalPath`
2. **`flyer_items/<shard>/*.png`** — cropped item thumbnails generated by the parser, referenced by `FlyerItem.LocalPhotoPath`

Both directories grow indefinitely. With the scheduler currently off, the live dataset is 7 GB (`flyer_items`) + 1.3 GB (`uploads/flyer_pages`), and each daily backup re-packs all of it (~7.9 GB/day × 10 retention = 79 GB). The DB itself is 11 MB.

Flyers are weekly promotions. Once a flyer ends, its images have no operational purpose. Historical price data in the DB is valuable; the image files are not.

## Goals / Non-Goals

**Goals:**
- Delete local image files for flyers that ended more than 30 days ago
- Clear `FlyerPage.LocalPath` and `FlyerItem.LocalPhotoPath` in DB after deletion
- Run cleanup before each daily backup so backups shrink immediately
- Degrade gracefully in the UI: show placeholder when local image is absent

**Non-Goals:**
- Deleting DB records (flyers, pages, items, prices, dates are preserved forever)
- Clearing `FlyerItem.PhotoURL` (remote URL is preserved but not used as a UI fallback)
- Changing backup retention count or schedule
- Cleaning up files not tracked by DB records (orphan sweep is a separate concern)

## Decisions

### D1: Run cleanup as the first step of `backup.Task.run()`

**Chosen:** Hook into the existing `backup.Task` before the archive is created.

**Rationale:** The backup already runs daily, is already scoped by `dataPath`, and has access to `dbPath`. Adding a pre-archive step means the backup immediately benefits from the freed space. A separate goroutine would require new lifecycle management for negligible gain.

**Alternative considered:** Separate goroutine with its own ticker. Rejected — two schedulers for one concern, no benefit over piggyback approach.

---

### D2: Eligibility based on `Flyer.EndDate`, fallback to `Flyer.CreatedAt + 30 days`

**Chosen:** A flyer is eligible when `MAX(EndDate, CreatedAt) + 30 days < now`.

If `EndDate` is zero (unset), treat `CreatedAt` as the reference. This handles flyers created by `DownloadNewFlyers` before parsing sets dates.

**Alternative considered:** Using `FlyerItem.EndDate` per item. Rejected — items within a flyer share the same lifecycle; flyer-level expiry is simpler and correct.

---

### D3: Delete files then clear DB paths in a single pass per flyer

**Chosen:** For each eligible flyer:
1. Query all `FlyerPage` records with non-empty `LocalPath`
2. Delete each file; log errors but continue
3. Bulk-update `LocalPath = ""` for all pages of that flyer
4. Query all `FlyerItem` records with non-empty `LocalPhotoPath`
5. Delete each file; log errors but continue
6. Bulk-update `LocalPhotoPath = ""` for all items of that flyer

If a file is already missing, log a warning and proceed (idempotent).

**Alternative considered:** Transactional file+DB rollback. Rejected — SQLite + filesystem cannot be made truly atomic; best-effort with logging is sufficient. A re-run will clear any DB paths whose files were already deleted.

---

### D4: UI graceful degradation — placeholder only, no PhotoURL fallback

**Chosen:** When `LocalPhotoPath` is empty, the flyer card shows the existing placeholder image. `PhotoURL` is not used as a fallback src.

**Rationale:** `PhotoURL` points to a retailer's CDN that may rotate or expire independently. Relying on it would introduce unpredictable image availability. A consistent placeholder is simpler and more reliable.

**Alternative considered:** Fall back to `PhotoURL` before showing placeholder. Rejected — adds complexity and depends on third-party URL stability.

## Risks / Trade-offs

- **Partial deletion on crash** → Mitigation: cleanup is idempotent; next run clears remaining files and DB paths
- **Re-parse requires re-download** → If someone wants to re-parse old flyers, source pages will be gone. Mitigation: `FlyerPage.SourceURL` is preserved — pages can be re-downloaded from shop URLs (while the retailer still hosts them)
- **30-day window is fixed** → Not configurable via env var in v1. Can be added later if needed
- **Large initial cleanup** → First run will delete ~8 GB. Backup after first cleanup will be much smaller. No user impact

## Migration Plan

1. Deploy new backend with cleanup step enabled
2. On first startup: backup task runs cleanup (deletes ~8 GB of images for all flyers with EndDate > 30 days ago), then creates a small backup archive
3. Old backups cycle out over the next 10 days; full 79 GB reclaimed automatically
4. No DB migration needed; `LocalPath`/`LocalPhotoPath` are already string fields defaulting to empty

**Rollback:** Revert to previous backend image. Deleted files cannot be recovered, but the DB records are intact. Re-parsing would regenerate item crops (Gemini API cost).
